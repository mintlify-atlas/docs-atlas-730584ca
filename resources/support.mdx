## How to File Issues

This project uses GitHub Issues to track bugs and feature requests.

<Steps>
  <Step title="Search existing issues">
    Before filing a new issue, please search the [existing issues](https://github.com/microsoft/kanagawa/issues) to avoid duplicates.
  </Step>

  <Step title="Create a new issue">
    If you can't find an existing issue, [create a new issue](https://github.com/microsoft/kanagawa/issues/new) with:

    <Checklist>
      - [ ] Clear, descriptive title
      - [ ] Detailed description of the problem or request
      - [ ] Steps to reproduce (for bugs)
      - [ ] Expected vs actual behavior
      - [ ] Kanagawa version and platform
      - [ ] Minimal code example (if applicable)
      - [ ] Error messages or logs
    </Checklist>
  </Step>

  <Step title="Engage with maintainers">
    Respond to questions and feedback from maintainers to help resolve the issue.
  </Step>
</Steps>

## Bug Reports

When reporting bugs, please include:

<AccordionGroup>
  <Accordion title="Environment Information">
    - Kanagawa version
    - Operating system (Linux, macOS, Windows)
    - Build configuration (compiler version, CMake version)
    - Any relevant environment variables
  </Accordion>

  <Accordion title="Reproduction Steps">
    Provide clear, step-by-step instructions to reproduce the issue:
    ```bash
    # Example reproduction steps
    1. Create a file with this code: [code example]
    2. Run: kanagawa compile file.kng
    3. Observe error: [error message]
    ```
  </Accordion>

  <Accordion title="Expected vs Actual Behavior">
    Clearly describe:
    - What you expected to happen
    - What actually happened
    - Any error messages or unexpected output
  </Accordion>

  <Accordion title="Minimal Example">
    If possible, provide a minimal code example that demonstrates the issue:
    ```kanagawa
    // Minimal example that reproduces the bug
    module example {
        // Your code here
    }
    ```
  </Accordion>
</AccordionGroup>

## Feature Requests

We welcome feature requests! When suggesting a new feature:

- Describe the use case and motivation
- Provide examples of how it would be used
- Explain why this feature would be valuable
- Consider potential alternatives
- Be open to discussion and feedback

## Response Times

<Warning>
  This project is maintained by volunteers. Response times may vary depending on maintainer availability and issue complexity.
</Warning>

- **Critical bugs** - Addressed as quickly as possible
- **Bug reports** - Reviewed within a few days to a week
- **Feature requests** - May take longer to discuss and implement
- **Questions** - Community members often respond quickly

## Microsoft Support Policy
